⚠️ Antes de empezar — verificar que ambos servidores estén corriendo
Backend en puerto 3001 (iniciado con npm run dev) y Frontend en puerto 5173 (iniciado con npm run dev desde la carpeta frontend). Si el backend muestra "app crashed", revisar la consola de nodemon para ver el error.

Conceptos clave del sistema

Fundamentos
Concepto Regla de negocio Dónde se valida
Duración del turno Siempre exactamente 1.5 horas (90 min). Ej: 10:00 → 11:30. Nunca más, nunca menos. Backend (POST /turnos-fijos valida duracion === 90) + frontend genera slots de 90 min
Estado inicial reserva Reserva eventual del jugador → pendiente. Admin debe confirmar. Backend — POST /reservas siempre crea con estado: 'pendiente'
Estado inicial clase Clase del profesor → confirmada inmediatamente. No necesita aprobación. Backend — POST /reservas/profesor siempre crea con estado: 'confirmada'
Estado turno fijo Solicitud → pendiente. Admin aprueba → confirmado. Admin baja → inactivo. Backend — PATCH /turnos-fijos/:id/estado
Anti-colisión (TOCTOU) Dos jugadores no pueden reservar el mismo slot simultáneamente. Backend — operaciones dentro de $transaction con re-verificación atómica
Prioridad turno fijo Un turno fijo confirmado bloquea su horario para reservas eventuales del mismo día. Backend — al crear reserva eventual, verifica TF activos sin ausencia en esa fecha
ClubId aislado El clubId siempre viene del JWT del usuario, nunca de un parámetro de URL o body. Backend — req.user.clubId en todos los endpoints. Garantiza aislamiento multi-tenant.
Zona horaria Toda la lógica de "el turno ya comenzó" usa UTC-3 (Argentina, sin DST). Backend — new Date(Date.now() - 3 * 60 * 60 * 1000)

Bloque 1 — Reserva eventual (Jugador)

Jugador + Admin

Una reserva eventual es una reserva de cancha para un día específico. El jugador elige fecha, cancha y horario. El sistema valida disponibilidad en tiempo real y crea la reserva en estado pendiente hasta que el admin la apruebe.

1.1
Jugador — Navegar a Reservar una Cancha
/dashboardJugadores/reservas
▶
  1. Iniciá sesión como jugador en /dashboardJugadores.
  2. En el sidebar, hacé clic en "Reservar cancha".
  3. Al entrar, notá el banner colapsable "¿Cómo funciona esta sección?" en la parte superior. Hacé clic para expandirlo — explica la diferencia entre reserva puntual y turno fijo semanal.
  4. Verificá que la pantalla muestra un selector de fecha (semana actual) y una grilla de canchas × horarios.
  5. Al pie de la página podés ver dos banners compactos de acceso rápido: uno a "Mis reservas" (eventuales próximas) y otro a "Mis turnos fijos" (activos y pendientes). Son links directos a cada sección.
✅ Resultado esperado
La grilla muestra las canchas del club (columnas) y los horarios disponibles en bloques de 1.5h. Los slots ya ocupados aparecen bloqueados (sombreados). Los slots disponibles son interactuables. El helper informativo está cerrado por defecto para no estorbar a usuarios que ya conocen el sistema.
💡 Por qué funciona así
Al montar la página, el frontend llama a GET /api/reservas?fecha=HOY y GET /api/turnos-fijos/slots-ocupados. Con ambas respuestas construye la grilla. Los horarios disponibles se calculan con generarSlots() usando la apertura/cierre del club desde el clubStore.
1.2
Jugador — Seleccionar fecha, cancha y horario
/jugadores/reservas — grilla
▶
  1. Usá las flechas del selector para ir a mañana (evitamos hoy para no chocar con la restricción de hora pasada).
  2. Hacé clic sobre un slot verde (disponible) en cualquier cancha.
  3. Aparece un panel de confirmación a la derecha (o un modal en mobile) con los detalles: cancha, fecha, inicio, fin, precio.
  4. Verificá que la hora de fin es siempre inicio + 1.5 horas.
  5. Hacé clic en "Confirmar reserva".
  6. (Opcional) Activá el toggle "Turno fijo" antes de confirmar para que la reserva sea recurrente semanal. El subtítulo bajo el toggle cambia para indicar a dónde irá cada tipo: "Solo para este día · Lo verás en Mis reservas" (desactivado) o "Se repetirá cada semana · Lo gestionás en Mis turnos fijos" (activado).
🔒 Validaciones del backend (POST /api/reservas)
El slot no debe superponerse con reservas existentes (pendientes o confirmadas).
El slot no debe superponerse con un turno fijo confirmado (sin ausencia en esa fecha).
Si es hoy, la hora de inicio no puede ser anterior o igual a la hora actual (UTC-3).
Cancha debe existir y estar activa en el club del jugador.
El clubId viene del JWT del jugador — no del frontend.
✅ Resultado esperado
La reserva se crea con estado pendiente. En la grilla, el slot queda sombreado con color "pendiente" (amarillo/naranja). Aparece un toast verde/lima fijo en la esquina superior derecha con el mensaje: "📅 Reserva enviada · [Cancha] [Horario] — El admin la revisará". Ese toast no tiene timer — persiste hasta que el admin apruebe la reserva (en ese momento desaparece automáticamente). Si en el mismo turno también se reservó un turno fijo, ambos toasts coexisten: el verde para la reserva eventual y un toast ámbar para el turno fijo.
❌ Si el slot ya está tomado
El backend responde 409 Conflict con el mensaje "El horario ya está reservado" o "Ese horario pertenece a un turno fijo reservado". La UI muestra un error y no crea la reserva.
🔔 Notificación generada
Al admin del club: tipo nueva_reserva con nombre del jugador, cancha, fecha y horario.
1.3
Jugador — Verificar que la reserva aparece en "Mis reservas"
/dashboardJugadores/mis-reservas (ítem propio en el sidebar)
▶
  1. En el sidebar del jugador, hacé clic en "Mis reservas" (ícono ClipboardList, entre "Reservar cancha" y "Mis turnos fijos").
  2. La reserva recién creada debe aparecer en el tab Próximas con badge PENDIENTE en amarillo.
  3. Desde esta pantalla también podés cancelarla: el botón "⊗ Cancelar" abre un modal de confirmación con la política de horasCancelacion del club.
  4. La grilla en "Reservar cancha" sigue mostrando el slot ocupado (indicador visual de tu propia reserva).
✅ Resultado esperado
La reserva aparece en el listado con estado pendiente. El jugador NO puede ver datos de otros jugadores en la grilla (solo sabe que el slot está ocupado, no por quién — privacidad implementada en el backend).
💡 Por qué el jugador no ve los nombres de otros
En GET /api/reservas, cuando el rol es jugador, el include de jugador es false. Solo el admin recibe nombre y apellido de quien reservó.
1.4
Admin — Ver y aprobar la reserva pendiente
http://localhost:5173/dashboard → Grilla / Notificaciones
▶
  1. Iniciá sesión como admin en /login.
  2. En la sección "Grilla de reservas", el panel "Avisos de jugadores" muestra la notificación de nueva reserva. Si hay muchas, el panel tiene scroll interno (máx. ~4 filas visibles).
  3. Si hay más de una reserva pendiente, aparece el botón "Aprobar todas (N)" en el encabezado del panel — aprueba todas las pendientes en un solo clic.
  4. Para aprobar de a una: hacé clic en el botón "Confirmar" de la fila correspondiente.
  5. Alternativamente, hacé clic sobre el slot amarillo en la grilla y confirmá desde el panel lateral.
🔒 Validaciones al confirmar (PATCH /api/reservas/:id/estado)
La reserva debe pertenecer al club del admin (verificado por clubId del JWT).
Re-verifica que el slot no esté ocupado por otra reserva (pendiente o confirmada). Protege el caso donde el admin cancela una reserva, otro jugador ocupa el slot, y luego el admin intenta reactivar la primera.
La re-verificación + update ocurren dentro de $transaction para evitar race conditions.
✅ Resultado esperado
El slot cambia a color verde confirmada en la grilla del admin. El jugador, al refrescar su grilla, también verá el slot confirmado. El toast verde/lima que mostraba "Reserva enviada…" en la vista del jugador desaparece automáticamente al detectar que el estado cambió a confirmada (sin necesidad de refrescar la página).
🔔 Notificación generada
Al jugador: notificación de reserva confirmada (implementado vía store, en backend real será via tabla notificaciones).

Bloque 2 — Turno Fijo (solicitud, aprobación, baja)

Jugador + Admin

Un turno fijo es una reserva recurrente semanal en el mismo día y horario. El jugador solicita el turno, el admin lo aprueba. Una vez confirmado, bloquea ese slot automáticamente en todos los futuros del mismo día de la semana.

2.1
Jugador — Solicitar turno fijo
http://localhost:5173/jugadores/turnos-fijos
▶
  1. Iniciá sesión como jugador y navegá a /jugadores/turnos-fijos.
  2. Hacé clic en "Solicitar turno fijo" (botón principal).
  3. Completá el formulario:
Día: Lunes (o cualquier día disponible)
Cancha: Cancha 1 (seleccionar del dropdown)
Horario: 10:00 — 11:30 (slots de 1.5h)
  1. Hacé clic en "Enviar solicitud".
🔒 Validaciones del backend (POST /api/turnos-fijos)
La duración debe ser exactamente 90 minutos. Si enviás 10:00 → 11:00, el backend responde 400: "El turno fijo debe durar exactamente 1.5h".
No puede existir ya un TF pendiente o confirmado en la misma cancha, día y horario superpuesto (RN-51). Si existe, responde 409.
La validación es atómica (dentro de $transaction) para evitar race conditions.
✅ Resultado esperado
Aparece el turno en la lista con badge PENDIENTE. Muestra día, horario y cancha. Un cartel explica que está esperando aprobación del admin.
🔔 Notificación generada
Al admin: tipo solicitud_turno_fijo con datos del jugador, cancha, día y horario.
2.2
Admin — Ver solicitudes pendientes y aprobar
/dashboardAdmin/reservas → panel "Avisos de jugadores" → tab Turnos fijos
▶
  1. En el panel admin, el banner "Avisos de jugadores" muestra la notificación de tipo Solicitud turno fijo con el nombre del jugador, cancha y horario.
  2. Opción rápida (recomendada): hacé clic directamente sobre la notificación. El sistema la marca como leída automáticamente y te lleva al tab "Turnos fijos" sin ningún paso extra.
  3. Opción manual: hacé clic en la tab "Turnos fijos" en la parte superior de la grilla, luego aplicá el filtro "Pendientes".
  4. Localizá la solicitud del jugador (muestra nombre, cancha, día y horario).
  5. Hacé clic en "Aprobar" (botón verde).
🔒 Validaciones al aprobar (PATCH /api/turnos-fijos/:id/estado → "confirmado")
Re-verifica que no haya otro TF confirmado en la misma cancha+día+horario (otro jugador pudo haber sido aprobado mientras tanto).
Verifica que no haya reservas eventuales confirmadas o pendientes en las próximas 8 ocurrencias de ese día.
Si hay conflicto, responde 409 con detalle de qué fecha tiene el conflicto.
Toda la re-verificación + update ocurre dentro de $transaction.
✅ Resultado esperado
El turno pasa a estado confirmado en el panel. En la grilla de reservas del admin, ese slot aparece bloqueado todos los futuros del mismo día de la semana.
❌ Si hay conflicto con reserva eventual
El backend responde 409 con: "El horario tiene una reserva eventual confirmada el YYYY-MM-DD. Resolvé ese conflicto antes de confirmar el turno fijo." El admin debe primero cancelar esa reserva eventual y luego volver a aprobar.
🔔 Notificación generada
Al jugador: tipo turno_fijo_confirmado — "Tu turno fijo del [día] de [hora] en [cancha] fue aprobado".
2.3
Jugador — Verificar turno confirmado y próxima fecha
/jugadores/turnos-fijos
▶
  1. Volvé a la sección Turnos Fijos del jugador.
  2. El turno ahora muestra badge CONFIRMADO en verde.
  3. Verificá que muestra la próxima fecha calculada: si hoy es el día del turno y la hora no pasó → muestra hoy; sino → el siguiente miércoles/lunes/etc.
  4. En la grilla de reservas (/jugadores/reservas), verificá que ese slot del día de la semana aparece bloqueado.
✅ Resultado esperado
El turno fijo confirmado bloquea automáticamente el slot en la grilla de reservas para todas las ocurrencias futuras. Si otro jugador intenta reservar ese horario el mismo día de la semana, el sistema lo rechazará.
💡 Por qué se bloquea en la grilla
Al cargar la grilla, el frontend llama a GET /api/turnos-fijos/slots-ocupados que retorna todos los TF confirmados del club (sin datos personales). La función generarSlots() excluye esos horarios del día de la semana correspondiente, a menos que el TF tenga marcada una ausencia para esa fecha específica.
2.4
Admin — Rechazar una solicitud (alternativa)
/dashboard → Turnos Fijos → filtro Pendientes
▶
  1. Para probar el rechazo, solicitá un segundo turno fijo con el jugador (en otro horario disponible).
  2. En el panel admin, localizá esa segunda solicitud.
  3. Hacé clic en "Rechazar" (botón rojo).
✅ Resultado esperado
El turno pasa a estado inactivo. El jugador recibe una notificación de rechazo y el turno desaparece de su lista activa (los estados inactivos no se muestran).
🔔 Notificación generada
Al jugador: tipo turno_fijo_rechazado — "Tu solicitud de turno fijo fue rechazada".
💡 Diferencia entre rechazar y dar de baja
Si el TF estaba en pendiente → la notificación es turno_fijo_rechazado. Si el TF estaba en confirmado → la notificación es turno_fijo_baja. El backend detecta el estado previo antes de hacer el update.

Bloque 3 — Clases del Profesor

Profesor + Admin

El profesor tiene su propio portal de agenda. Puede registrar clases directamente — que se crean como confirmadas inmediatamente sin aprobación del admin. El admin también puede crear clases en nombre de cualquier profesor desde su panel.

3.1
Profesor — Login y verificar portal de agenda
http://localhost:5173/profesores/login
▶
  1. Navegá a http://localhost:5173/profesores/login.
  2. Ingresá con email y password del profesor. El campo clubId se resuelve automáticamente desde la variable de entorno VITE_CLUB_ID.
  3. Al hacer login, el backend firma un JWT con role: 'profesor' y clubId.
  4. Al montar el layout, el frontend llama a GET /auth/profesor/me para restaurar el objeto profesor completo (esto permite que al refrescar la página no se pierda la sesión).
  5. Verificá que llegás al portal /profesores/agenda y que se muestra tu nombre en el header.
✅ Resultado esperado
La agenda del profesor muestra un selector de fecha y la grilla de sus clases del día. Si el profesor tiene canchas asignadas en su perfil (canchasIds), solo ve esas canchas. Si no tiene asignación → ve todas las canchas activas del club.
❌ Si el login falla con "Failed to fetch"
El backend no está corriendo. Iniciarlo con npm run dev desde project/apps/backend/.
3.2
Profesor — Registrar una clase en su agenda
/profesores/agenda
▶
  1. En la agenda del profesor, seleccioná mañana como fecha.
  2. Hacé clic en "+ Nueva clase" o en un slot disponible en la grilla.
  3. Completá el formulario:
Cancha: Cancha 1
Horario: 14:00 — 15:30 (1.5h)
Precio: 2000 (opcional)
Descripción: Clase grupal principiantes (campo se llama "Descripción" cuando tipo = clase)
  1. Hacé clic en "Crear clase".
🔒 Validaciones del backend (POST /api/reservas/profesor)
El profesor debe tener asignada esa cancha (o no tener ninguna asignada, en cuyo caso puede usar cualquiera).
No puede superponerse con reservas existentes (pendientes o confirmadas) en esa cancha y fecha.
No puede superponerse con turnos fijos confirmados activos.
El mismo profesor no puede tener otra clase en ese horario en otra cancha.
La clase no puede haber comenzado ya (UTC-3).
✅ Resultado esperado
La clase aparece inmediatamente en la grilla de la agenda del profesor con estado confirmada (color azul/morado). No requiere aprobación del admin.
💡 Por qué las clases se confirman sin aprobación
Los profesores son usuarios confiables del club (dados de alta por el admin). Por diseño del negocio, sus clases tienen confirmación inmediata. La reserva se crea con tipo: 'clase' y estado: 'confirmada' directamente en el endpoint POST /reservas/profesor.
3.3
Admin — Ver la clase del profesor en la grilla general
/dashboard → Grilla de reservas
▶
  1. Iniciá sesión como admin.
  2. Navegá a la Grilla de reservas y seleccioná la misma fecha donde el profesor creó su clase.
  3. Verificá que el slot de la clase aparece en la grilla con el nombre del profesor visible.
✅ Resultado esperado
La clase del profesor es visible en la grilla del admin con el nombre del profesor. El slot está bloqueado para reservas eventuales (los jugadores no pueden reservar ese horario).
💡 Por qué la clase aparece en la grilla del admin
El endpoint GET /api/reservas retorna todas las reservas del club (sin filtrar por tipo). Incluye clases de profesores (tipo: 'clase'), reservas eventuales y bloqueos manuales. El include de profesor siempre se aplica para cualquier rol, así el admin puede ver de quién es cada clase.
3.4
Admin — Crear clase en nombre de un profesor (Tab Clases)
/dashboard → Grilla → Tab "Clases"
▶
  1. En el panel admin, dentro de la Grilla de reservas, buscá el tab "Clases".
  2. Seleccioná un profesor del dropdown.
  3. Elegí cancha, fecha y horario.
  4. Hacé clic en "Crear clase".
🔒 Validaciones del backend (POST /api/reservas/admin/clase-profesor)
El profesor debe pertenecer al mismo club que el admin.
Mismas validaciones de colisión que cuando el profesor crea directamente.
Si el profesor ya tiene clase en ese horario en otra cancha, se rechaza con 409.
✅ Resultado esperado
La clase aparece en la grilla del admin inmediatamente. Al refrescar la agenda del profesor (/profesores/agenda), la clase también aparece ahí.
❌ Bug conocido — resuelto en esta versión
En versiones anteriores, la clase creada por el admin no aparecía en la grilla inmediatamente porque se usaba addReservaAdmin() con el objeto crudo del backend (que tiene horaInicio/horaFin en vez de inicio/fin). El fix: ahora se llama a fetchReservasBackend() que re-fetcha y re-mapea correctamente.
3.5
Profesor — Eliminar una clase propia
/profesores/agenda
▶
  1. En la agenda del profesor, hacé clic sobre una clase existente.
  2. Aparece un panel/modal con los detalles de la clase.
  3. Hacé clic en "Eliminar" (botón rojo).
  4. Confirmá la eliminación en el diálogo de confirmación.
🔒 Validaciones del backend (DELETE /api/reservas/:id)
El profesor solo puede eliminar sus propias clases (reserva.profesorId === req.user.id).
Si intenta eliminar la clase de otro profesor → 403 Forbidden.
✅ Resultado esperado
La clase desaparece de la agenda del profesor y el slot queda libre en la grilla del admin (dentro del intervalo de polling de 30 segundos, o inmediatamente si refresca).
3.6
Admin — Gestionar perfil de un profesor (celular, contraseña, activar/desactivar)
/quienes-somos → sección Profesores
▶
  1. Iniciá sesión como admin y navegá a /quienes-somos.
  2. En la sección Profesores, hacé clic en el ícono de lápiz ✏️ en la card de un profesor.
  3. Verificá que el formulario muestra: Nombre, Apellido, Email, campo Contraseña (vacío = sin cambio), Celular (tel), Especialidad, toggle Acceso al portal.
  4. Modificá el celular y hacé clic en "Guardar". Verificá que el cambio persiste.
  5. Para resetear la contraseña: escribí la nueva en el campo y guardá. Campo vacío → hash existente no se toca.
  6. Para desactivar: desactivá el toggle "Acceso al portal" y guardá. El profesor no podrá iniciar sesión.
✅ Resultado esperado
La card del profesor se actualiza con los nuevos datos. Acceso desactivado → error de autenticación al login. Celular guardado vía PATCH /api/profesores/:id.
3.7
Profesor — Autocompletar genera clases de 1 hora separadas
/profesores/agenda → botón "Autocompletar"
▶
  1. Iniciá sesión como profesor y navegá a /profesores/agenda.
  2. Seleccioná un día con varias horas libres (por ej. un martes con horario 09:00–18:00 y alguna clase ya cargada).
  3. Hacé clic en "Autocompletar".
  4. Verificá que el modal de confirmación muestra bloques individuales de 1 hora (no un único bloque que abarca todo el día). Por ej.: 09:00–10:00, 10:00–11:00, 11:00–12:00, etc.
  5. Confirmá. Verificá que se crean N reservas separadas, cada una de 1h, en distintos slots de la grilla.
🔒 Comportamiento esperado de calcularBloquesFaltantes
La función divide el tiempo libre en chunks de DURACION_CLASE = 60 minutos.
Cada chunk es un bloque independiente — se pueden aprobar/rechazar por separado.
Si la franja libre es de 2.5h (por ej. 09:00–11:30), genera 2 clases: 09:00–10:00 y 10:00–11:00 (el medio hora restante no alcanza para otro bloque).
Eliminar una clase solo borra esa reserva puntual — las demás clases del día no se tocan.
✅ Resultado esperado
Autocompletar llena el horario con clases individuales de 1h. La grilla del admin muestra cada clase en el slot correcto. Eliminar una clase libera solo ese slot (no borra "todo el bloque").
❌ Si falla (bug anterior ya corregido)
Si aparece un único bloque gigante que ocupa toda la tarde (ej. 10:00–18:00): el algoritmo está usando el extremo del segmento libre como blockEnd en lugar de iterar en chunks de 60 min. Verificar que calcularBloquesFaltantes usa el bucle while (cur + DURACION_CLASE <= segTo).
3.8
Profesor — "Esta semana" detallado y panel de ayuda (HelpCircle)
/profesores/agenda
▶
  1. En la agenda del profesor, buscá la sección "Esta semana".
  2. Verificá que los días con clases muestran chips agrupados por cancha:

Bloque 8 — Prueba real con dos jugadores simultáneos

Jugador × 2

Esta prueba valida que el sistema protege correctamente contra race conditions cuando dos jugadores intentan reservar el mismo slot al mismo tiempo. Se usa una ventana normal + una ventana de incógnito (o dos perfiles de Chrome distintos) para tener dos sesiones JWT independientes.

⚙️ Setup: cómo abrir dos sesiones
  • Ventana A (Jugador 1): Navegador normal → http://localhost:5173/jugadores → login con el jugador principal.
  • Ventana B (Jugador 2): Ventana de incógnito (Ctrl+Shift+N en Chrome) → misma URL → login con un segundo jugador distinto.

8.1
Doble reserva en el mismo slot — validación de atomicidad
Ventana A + Ventana B → /jugadores/reservas
▶
  1. En ambas ventanas, navegá a la misma fecha futura y elegí la misma cancha.
  2. Verificá que ambas grillas muestran el mismo slot como disponible (blanco/libre).
  3. En Ventana A: abrí el modal de reserva para ese slot pero no confirmes todavía.
  4. En Ventana B: abrí el modal del mismo slot.
  5. Hacé clic en "Confirmar" en Ventana B primero — esperá el toast de éxito.
  6. Ahora hacé clic en "Confirmar" en Ventana A (el slot ya fue tomado por B).
🔒 Lo que el backend debe hacer
La primera reserva (B) se procesa: POST /api/reservas verifica disponibilidad y crea la reserva en una transacción.
La segunda reserva (A) llega después: el backend detecta el conflicto y responde con 409 Conflict.
El frontend de A muestra un toast de error: "El horario ya no está disponible".
La reserva de A no se crea — no queda registro huérfano en la base de datos.
✅ Resultado esperado
Solo una reserva existe en la base de datos para ese slot. El jugador de Ventana A recibe error claro. Al refrescar la grilla en Ventana A, el slot aparece ocupado (gris/bloqueado).
❌ Si falla
Si ambas reservas se crean: hay una race condition real. Verificar que el endpoint POST /api/reservas hace la comprobación de disponibilidad dentro de una transacción Prisma (prisma.$transaction()) y no en dos queries separadas.
8.2
Grilla se actualiza sin refresco manual — polling entre ventanas
Ventana A + Ventana B → /jugadores/reservas
▶

  2. En Ventana A reservá un slot cualquiera (que esté libre).
  3. Sin refrescar, esperá hasta 30 segundos en Ventana B.
  4. Observá si el slot reservado por A aparece bloqueado en la grilla de B.
✅ Resultado esperado
En ≤30 segundos, la grilla de Ventana B muestra el slot como ocupado (gris), sin que el jugador B haya refrescado manualmente. Esto confirma que el polling de fetchReservasBackend() cada 30 seg está activo.
💡 Por qué existe el polling
El frontend no usa WebSockets. Para que la grilla refleje cambios de otros jugadores, PlayerReservasPage hace un fetch periódico al backend cada 30 segundos y actualiza el estado local. La grilla es reactiva a ese estado.

Ventana B → /jugadores/reservas
▶

  3. Observá el slot de 10:00–11:30 en Cancha 1 — debe aparecer bloqueado (gris, no clickeable).
